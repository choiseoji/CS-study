# 최악의 시간복잡도를 가지는 경우 해결방안은 무엇인가요?

사실 어떤 알고리즘을 쓰느냐에 따라 선택이 달라지겠지만, 여기서는 대부분 경우에서 선택지로 존재하는 것을 먼저 말하고 예시로 알아보자.

## 해결 방법

1. **자료 구조 변경**

O(N)의 탐색 시간이 걸리는 배열, 리스트 대신에 해시 함수를 통해 `O(1)` 만에 데이터를 찾을 수 있는 `HashMap` 이나 `HashSet` 으로 데이터를 옮긴다.

2. **알고리즘 교체**

퀵 정렬은 최악의 경우 `O(n^2)` 이 걸리기에 비슷한 시간 복잡도이지만 일정한 성능인 병향 정렬이나 힙 정렬로 교체한다.

3. **중복 연산 고려**

만약 피보나치 수열 같이 하위 연산이 동일하게 들어가는데 이가 엄청 많이 반복되어 있는 경우는 한 번 계산한 결과를 메모리 등에 저장해 활용한다.

4. **캐싱**

이 알고리즘, 자료구조 라는 것도 데이터와 관련된 내용이기에 당연히 캐싱을 적용할 수 있다. 만약 DB 에서 쿼리 속도가 느리면 캐싱을 고려하는 것처럼 이를 통해 추후 연산 자체를 건너뛰게 할 수 있다.

- 단 위의 경우 알고리즘에서는 잘 고민해야 하는 부분이 오히려 많이 중복되는 것이 아닌데 캐시에 올리게 되면 캐시 올리는 시간 + 어짜피 미스나서 다시 찾아야 함 .. 등등의 시간으로 안 쓰는 것보다 별로다.

***

## 구체적인 예시

### 1. 버블 정렬

- 데이터가 역순으로 정렬되어 있을 때 매번 스왑이 발생함.

우리가 계산하는 시간 복잡도는 크지는 않다면 조금 더 계산이 들어갈 것임.

이 경우는 알고리즘을 교체하는 것이 더 이득. -> 어짜피 잘 안스기에


### 2. 병합 정렬

- 항상 `O(nlogn)`을 보장하지만, 공간 복잡도가 높아짐.

그래서 이는 앞에서 말한 TimSort 를 사용해서 데이터 중 이미 정렬된 구간을 미리 파악해서 불필요하게 쪼개는 것을 생갹하고 병합 시에도 필요 메모리를 최소화할 수 있음.

### 3. 힙 정렬

- 항상 `O(nlogn)`을 보장하지만, CPU 캐시 히트율이 낮아짐.

그래서 퀵 sort 를 더 선호하는 것임..


### 4. 퀵 정렬

- 앞에서 말했듯 피벗을 최소/최대로 계속해 잡으면 `O(n^2)` 이 됨.

피벗을 중간값으로 고르게 하거나 난수로 함.

퀵 정렬은 또한 안정적인 정렬이 아니지만 앞서 말한 정렬 중 가장 효율적이기에 재귀 깊이 임계치를 설정해 이를 넘어가면 최악의 상황으로 판단해 힙 정렬로 스위칭 할 수 있도록 코드를 작성해놓음.


### 5. 트리 정렬

- 이는 자식 규칙이 있기에 완전 트리가 아니어서 만약 오름차순으로 정렬된 데이터를 계속 넣으면 편향 트리가 됨. (한쪽이 엄청 길어짐)

결론적으로 위의 경우 탐색 및 정렬 시간이 `O(n^2)` 이 됨.

이 때는 자가 균형 트리를 선택하는 것이 이득임. (뒤에 누군가가 발표할 레드-블랙 트리)
