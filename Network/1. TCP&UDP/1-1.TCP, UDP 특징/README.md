## TCP와 UDP 특징과 차이점에 대해 간략히만 설명해주세요.
> TCP는 신뢰성, UDP는 속도와 실시간성에 초점을 둔 전송 계층 프로토콜입니다.
>
> TCP는 연결을 맺고 데이터를 순서대로 정확히 전달합니다. 손실된 데이터는 재전송하므로 신뢰성이 높지만 상대적으로 느립니다.
>
> UDP는 연결 없이 데이터를 바로 전송합니다. 순서 보장이나 재전송은 없지만 빠르고 지연이 적습니다.
> 
> 즉, TCP는 파일 전송·웹처럼 정확성이 중요할 때, UDP는 게임·영상처럼 빠른 전송이 중요할 때 사용됩니다.

### 전송 계층
송신자와 수신자를 연결하는 통신 서비스를 제공하는 계층

-> 데이터 전달을 위해 사용하는 프로토콜: TCP, UDP

### TCP
인터넷상에서 데이터를 메세지의 형태로 보내기 위해 IP와 함께 사용하는 프로토콜

![](https://img1.daumcdn.net/thumb/R1280x0/?scode=mtistory2&fname=https%3A%2F%2Fblog.kakaocdn.net%2Fdna%2Fod4Bz%2FbtrP3MyRXko%2FAAAAAAAAAAAAAAAAAAAAAGj9JpGiI_O7yN-04ZN2vHP_cPdevKNIyUBfFJ7ULl3h%2Fimg.png%3Fcredential%3DyqXZFxpELC7KVnFOS48ylbz2pIh7yKj8%26expires%3D1782831599%26allow_ip%3D%26allow_referer%3D%26signature%3DkpTQmOZ3PcYAHB4zrs9x3UI94MA%253D)

#### TCP 특징
- 연결 지향 방식으로 패킷 교환 방식 사용 (가상 회선 방식)
- 3-way Handshake로 연결 설정 & 4-way Handshake로 연결 해제
- 흐름 제어 및 혼잡 제어
- 높은 신뢰성
- UDP보다 속도 느림
- Full-Duplex, Point to Point 방식

#### TCP는 연속성보다 신뢰성 있는 전송이 중요할 때 사용

#### TCP 서버 특징
- 서버 소켓은 연결만 담당
- 연결 과정에서 반환된 클라이언트 소켓은 데이터 송수신에 사용
- 서버와 클라이언트는 1:1로 연결
- 스트림 전송으로 전송 데이터의 크기가 무제한
- 패킷에 대한 응답을 해야하기 때문에 (시간 지연, CPU 소모) 성능이 낮음
- Streaming 서비스에 불리 (손실된 경우 재전송 요청)


### UDP
데이터를 데이터그램 단위로 처리하는 프로토콜

![](https://img1.daumcdn.net/thumb/R1280x0/?scode=mtistory2&fname=https%3A%2F%2Ft1.daumcdn.net%2Fcfile%2Ftistory%2F9969973359FEB59309)
#### UDP 특징
- 비연결형 서비스 (데이터그램 방식)
- 정보를 송수신할 때 신호절차를 거치지 않음
- UDP 헤더의 CheckSum 필드를 통해 최소한의 오류 검출
- 신뢰성이 낮음
- TCP보다 빠름

#### UDP는 신뢰성보다는 연속성이 중요한 서비스에 이용

#### UDP 서버 특징
- 연결 자체가 없어서 서버 소켓과 클라이언트 소켓 구분 없음
- 소켓을 활용해 IP와 PORT 기반으로 데이터 전송
- 서버와 클라이언트는 1:1, 1:N, N:M 등으로 연결 가능
- 데이터그램 단위로 전송되며, 데이터그램 최대 크기를 초과하면 잘라서 보냄
- 흐름 제어가 없어 순서가 맞는지 오류가 없는지 확인 불가
- 파일 전송과 같은 신뢰성이 필요한 서비스보다 성능 중요시 되는 경우 사용

### TCP VS UDP
| 구분     | TCP           | UDP            |
| ------ | ------------- | -------------- |
| 연결     | 연결 지향         | 비연결            |
| 데이터 방식 | 바이트 스트림       | 데이터그램          |
| 신뢰성    | 높음            | 낮음             |
| 순서 보장  | 보장            | 보장 안 함         |
| 재전송    | 있음            | 없음             |
| 흐름 제어  | 있음            | 없음             |
| 속도     | 상대적으로 느림      | 상대적으로 빠름       |
| 사용 예시  | 파일 전송, 웹, 이메일 | 게임, 음성/영상, DNS |


## TCP통신에서 지연 패킷이 발생했을 때 서버는 어떻게 처리하나요? (꼬리 질문)
> TCP에서 지연 패킷이 발생하면 Sequence Number로 순서를 확인해 재정렬하거나, 필요 없는 패킷은 폐기합니다.
>
> 예를 들어 3번 패킷이 2번보다 먼저 오면 서버는 3번을 버퍼에 임시 저장하고, 2번이 도착한 뒤 1, 2, 3 순서로 애플리케이션에 전달합니다.
>
> 만약 2번이 끝까지 오지 않으면 ACK를 통해 재전송을 유도합니다. 반대로 이미 처리한 패킷이 늦게 도착하면 중복 패킷으로 보고 버립니다.
>
> 즉, TCP는 지연 패킷을 바로 처리하지 않고 순서, 중복 여부, 연결 상태를 확인해 처리합니다.

### 1. 순서가 꼬인 경우
1번 패킷 -> 3번 패킷 -> 2번 패킷으로 도착한 경우!

TCP는 패킷마다 Sequence Number 가지고 있음!

```text
1번 처리
3번은 일단 버퍼에 저장
2번이 도착할 때까지 기다림
2번 도착
→ 1, 2, 3 순서로 애플리케이션에 전달
```

=> 늦게 온 패킷이라도 순서 번호를 보고 다시 맞춤

### 2. 패킷이 손실된 경우
아예 2번 패킷이 도착하지 않았다면...?

TCP는 2번 안오고 3번이 도착 -> 일정 시간후에도 안왔다면...?

ACK을 보냄 = 2번 보내줘~

```text
1. 일정 시간 동안 ACK가 안 오면 재전송
2. 같은 ACK가 반복되면 손실로 보고 빠르게 재전송
```

### 3. 너무 오래된 패킷
2번 패킷이 안와서 재전송을 한 상황

원래 2번 패킷은 네트워크 지연 중 / 재전송 2번 패킷은 서버에 먼저 도착

원래 2번 패킷이 그 이후 도착

```text
이건 이미 받은 데이터네?
→ 중복 패킷이므로 폐기
```

### 4. 연결과 매칭 실폐로 폐기
TCP는 패킷 받을 때 출발지 IP, 출발지 Port, 목적지 IP, 목적지 Port 확인

연결 종료 후 예전 패킷 도착한 상황

```text
이 패킷이 속한 TCP 연결이 아직 있나?
```

없으니까 폐기...

### 5. 새로운 견결과 혼란
TCP 연결 종료 후, 같은 클라이언트와 서버 다시 연결한 상황

그런데! 우연히 같은 포트 사용

재연결 후 옛날 옛적 보냈던 지연 패킷 도착 -> 새 연결 패킷으로 착각할 위험 존재

```text
그래서 TCP는 TIME_WAIT 상태로 일정 시간 유지

1. 네트워크에 남아 있는 예전 패킷이 사라질 시간을 줌
2. 예전 연결의 지연 패킷이 새 연결에 섞이는 것을 방지
3. 마지막 ACK가 손실되었을 때 재전송에 대응
```
