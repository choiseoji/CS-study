## Connection Timeout 과 Read Timeout 의 차이점은 무엇인가요?
> Connection Timeout은 서버와 TCP 연결을 맺는 단계에서 발생하는 타임아웃이고, Read Timeout은 연결은 성공했지만 서버 응답 데이터를 읽는 단계에서 발생하는 타임아웃입니다.
>
> 즉, Connection Timeout은 서버가 다운됐거나 방화벽, 포트 차단, backlog 포화 등으로 연결 자체가 일정 시간 안에 성립되지 않을 때 발생합니다.
> 
> 반면 Read Timeout은 TCP 연결은 정상적으로 맺어졌지만, 서버의 DB 조회, 외부 API 호출, GC, 대용량 응답 처리 등으로 인해 응답 데이터가 일정 시간 안에 오지 않을 때 발생합니다.

### Timeout 이란?
서버로 요청을 보냈지만 일정 시간 동안 답변을 받지 못하면 발생

-> 리소스 절약을 위함 (응답 시간이 길어지는 연결이 많으면 리소스가 모두 소진되어 장애로 이어질 수 있음)

### Connection Timeout
서버와 소켓 연결을 성공적으로 맺기까지 기다리는 최대 시간
![](https://miro.medium.com/v2/resize:fit:1100/format:webp/1*O92uZwwG65IWy5yTIZQEOw.png)

이 단계까지 일정 시간 내에 완료되지 않으면 Connection Timeout이 발생

#### 발생 원인
- 서버 다운
- 방화벽, 보안 그룹에서 port 막혀 있을 때
- 서버가 너무 바빠서 커넥션 수락 backlog가 꽉찼을 때

#### Connection Timeout이 길어지면....
- 서버가 다운됐어도 오랫동안 응답을 기다림
- 스레드가 오래 묶이면서 전체 시스템 병목 발생
- 장애 감지 속도 늦어짐

=> Fail Fast 원칙이 깨짐 (장애 시 빠른 차단을 위해 보통 1~2초로 설정)

### Read Timeout
클라이언트가 서버로부터 데이터를 읽는 I/O 작업 자체가 일정 시간 안에 완료되지 않을 때 발생

#### 작동 방식
전제: TCP 연결이 정상적으로 성립

1. 클라이언트는 서버 응답 읽음
2. 서버가 응답을 늦게 보내거나, 처리 시간이 길어져 한동안 데이터가 오지 않음
3. 설정한 Read Timeout 시간 안에 I/O 작업이 끝나지 않음
4. 클라이언트가 Read Timeout 발생 -> 연결 종료

=> 전체 읽기 작업 완료에 집중

#### 발생 원인
- 서버 응답이 너무 느림
- 서버의 내부 처리 (쿼리, I/O, GC)가 오래 걸림
- 대용량 응답 생성에 오래 걸림

=> 즉 연결은 정상이고 서버도 살아있는데 응답 I/O가 일정 시간 내에 완료 안되는 상황

### Socket Timeout
서버가 다음 패킷을 제때 보내지 않으면 발생

#### 작동 방식
1. 클라이언트와 서버의 TCP 연결이 완료
2. 서버가 데이터를 여러 패킷으로 나누어 전송
3. 클라이언트는 다음 패킷이 일정 시간 내에 도착하기를 기다름
4. 그 시간 제한이 socket timeout

#### 발생 원인
- 서버가 처리 도중 멈춰서 다음 패킷을 보내지 않을 때
- 서버의 네트워크 I/O가 막히거나 지연될 때
- Keep-Alive 중 일부 패킷이 유실되거나 지연될 때

=> 연결은 살아있지만 다음 패킷이 제때 도착하지 않는 문제
