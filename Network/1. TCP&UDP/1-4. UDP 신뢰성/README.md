## UDP는 항상 신뢰성을 보장하지 않나요?
> UDP 자체는 신뢰성을 보장하지 않습니다. UDP는 기본적으로 데이터를 빠르게 보내는 기능만 제공하고, 패킷 손실 여부 확인, 재전송, 순서 보장, 흐름 제어 같은 기능은 없습니다.
> 
> 다만 UDP 위에서 동작하는 애플리케이션이나 프로토콜이 이런 기능을 직접 구현하면 TCP와 비슷한 수준의 신뢰성을 가질 수 있습니다. 대표적인 예가 HTTP/3에서 사용하는 QUIC입니다. QUIC은 UDP 기반이지만 내부적으로 ACK, 재전송, 순서 보장, 흐름 제어, 혼잡 제어, TLS 암호화를 제공합니다.
> 
> 따라서 정리하면, UDP는 기본적으로 신뢰성을 보장하지 않지만, UDP 위에 구현된 프로토콜에 따라 신뢰성을 보장할 수 있습니다.
 

### UDP
UDP 자체는 신뢰성 보장 XX

UDP는 데이터 전송을 제외한 어떤 기능도 정의되어있지 않는 프로토콜

=> 개발자가 어플리케이션 구현을 어떻게 하냐에 따라 TCP와 비슷한 수준의 기능을 가질 수 있음

### HTTP/3 QUIC
QUIC은 UDP위에 TCP가 하던 기능을 애플리케이션 레벨에서 다시 구현한 프로토콜

```text
HTTP/2
애플리케이션: HTTP/2
보안: TLS
전송: TCP
네트워크: IP

HTTP/3
애플리케이션: HTTP/3
전송+보안: QUIC
기반: UDP
네트워크: IP
```
QUIC은 UDP 위에서 동작하지만, 내부적으로 스트림, 재전송, 흐름 제어, 혼잡 제어, TLS 기반 암호화를 제공

#### 신뢰성 보장 방식
UDP는 패킷을 그냥 던지고 끝이지만, QUIC은 각 데이터 조각에 번호를 붙이고 ACK를 관리

=> TCP 처럼 손실된 sequence를 다시 보냄 / 그러나 TCP처럼 연결 전체를 하나의 순서로 처리하는 것이 아니라 스트림별로 순서 보장

### QUIC의 기능
- 흐름 제어:
    
    받는 쪽이 처리 가능한 만큼만 보내도록 조절

- 혼잡 제어:

    네트워크가 혼잡해지면 전송 속도를 줄임

- 재전송 제어:

    손실된 데이터만 다시 보냄

- 순서 보장:

  스트림 내부 순서를 맞춰 애플리케이션에 전달

### 왜 굳이 UDP 위?
TCP는 운영체제 커널과 네트워크 장비에 깊게 들어가 있어서 TCP 변경을 위해서는...

```text
운영체제 수정
네트워크 장비 호환성 문제
방화벽, NAT, 프록시 영향
배포 속도 느림
```

### 주의 사항
```text
서버나 CDN이 HTTP/3를 제대로 지원하지 않는 경우
네트워크 손실이 거의 없는 안정적인 환경
QUIC 처리로 인한 CPU 오버헤드가 큰 경우
```
위와 같은 상황에서는 HTTP/3가 불리할수도 있음!

HTTP/3는 UDP 위에 QUIC을 올려 TCP의 신뢰성은 유지하면서, TCP의 구조적 지연 문제를 줄인 프로토콜
