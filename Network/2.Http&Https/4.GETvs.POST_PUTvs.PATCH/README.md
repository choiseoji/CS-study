# GET과 POST, PUT과 PATCH 의 차이는 무엇인가요?

***

> GET 은 읽을 때, POST 는 생성할 때 사용되는 메서드입니다. 이에 따라 GET은 멱등하고 POST는 멱등하지 않습니다.

> PUT는 리소스 전체 업데이트, PATCH는 리소스 부분 업데이트라는 차이점이 있습니다. 그렇기에 PUT은 멱등하지만 PATCH는 멱등하지 않는 것이 기본입니다.

### GET

HTTP 명세에 따르면 `GET` 요청은 오직 데이터를 읽을 때만 사용되고 수정할 때는 사용하지 않는다.

- 데이터의 변형의 위험이 있다면 사용하지 말라는 의미이다.

또한 `GET` 요청은 멱등(idempotent)하다.

- 같은 요청을 여러 번 하더라도 항상 같은 응답을 반환해야 한다.

***

### POST

HTTP 명세에 따르면 `POST` 요청은 주로 새로운 리소스를 생성할 때 사용한다.

- 데이터의 변경이 있다면 이를 사용하라는 의미이다.

또한 `POST` 요청은 멱등하지 않다.

- 같은 요청이라도 다른 응답을 반환할 가능성이 높다는 것이다.

***

### GET vs POST

일단 `POST` 는 요청 시 추가적인 데이터를 body 에 포함할 수 있다.
    즉, HTTP request 의 message body 에 나타난다.

반면 `GET`은 모든 필요 데이터를 URL 에 포함해 요청한다.
    즉, query string parameters 로 전달된다.

그 외의 차이점으로는 ...

| GET | POST |
| --- | --- |
| 파라미터들은 URL 일부이기에 브라우저 히스토리에 남음 | 안남음 |
| 파라미터가 URL로 인코딩 되기에 북마크 가능 | 불가 |
| 브라우저 캐시에 저장되어 있으면 다시 요청되지 않음 | 다시 요청됨 |
| 파라미터 데이터 제한 있음 | 파일도 업로드 가능 |
| 해킹되기 더 쉽다. | GET 에 비해 어려움 |
| ASCII 만 가능 | binary data 가능 |
| URL 이기에 길이 제한 존재 | 없다. |
| 모두에게 보여진다. | 보여지지 않는다. |

***

### PUT

`PUT` 은 요청 시 리소스의 정확한 URL을 알아야 한다.
    `/payments` 로 요청 시 `/payments/1` 같은 리소스가 생성되지 않는다.

또한 `PUT` 은 멱등하다.
    그렇기에 주로 업데이트 시 사용하지만 주의점은 **리소스 전체**를 업데이트해야 된다는 특징이 있다.

- 리소스 일부를 수정하고 싶어도 전체 필드 데이터를 요청 본문에 포함해야 한다.

### PATCH

`PATCH` 또한 업데이트 시 사용하는데 HTTP 명세에 따르면 이 또한 특정 리소스 URL 을 알아야한다.

그러나 이는 요청에 따라 리소스를 수정하고 **부분 업데이트**를 한다.

또한 멱등하지 않다.
    - RFC 문서에 따르면 `PATCH` 에 넣을 데이터가 `PUT` 에 넣을 데이터보다 크다면 PUT 이 더 적합하다고 기술하고 있다.

***

### POST vs PUT vs PATCH

***

| POST | PUT | PATCH |
| --- | --- | --- |
| 리소스 생성 또는 서버에서 정의한 작업 | 리소스 생성 및 전체 업데이트 | 리소스 부분 업데이트 |
| 리소스 컬렉션 URL | 특정 리소스 URL | 특정 리소스 URL |
| 멱등 X | O | X |
