## OS가 실행 주도권을 강제로 회수할 수 있는지에 따른 선점(Preemptive) 스케줄링과 비선점(Non-preemptive) 스케줄링의 차이는 무엇인가요?

***

> 선점 스케줄링은 OS가 통제권을 가지는 스케줄링으로 실행 중인 프로세스가 있더라도 OS가 판단하기에 더 급한 프로세스가 있거나 할단 시간이 끝나면 강제로 CPU를 회수해 다른 프로세스에 넘깁니다.
> 비선점형은 프로세스가 선택권을 가져 한번 CPU를 할당 받으면 프로세스 스스로 IO작업을 위한 대기나 실행 종료할 때까지 OS가 개입해 강제로 쫒아낼 수 없습니다.

***

### 선점형 스케줄링(Preemptive)

#### 📖 개념

- 하나의 프로세스가 CPU를 차지하고 있을 때, 우선순위가 높은 다른 프로세스가 현재 프로세스를 중단시키고 CPU를 점유하는 스케줄링 방식

#### 알고리즘

- 라운드 로빈(RR), SRT(Shortest Remaining Time First), 다단계 큐(Multi Level Queue), 다단계 피드백 큐(Multi Level Feedback Queue)

#### 메커니즘

1. 프로세스 A가 CPU를 **할당**받아 실행된다.

2. OS 타이머가 미리 정해진 시간 할당량 초과를 알리거나, 더 높은 우선순위 프로세스 B가 **Ready Queue** 에 들어온다.

3. OS가 개입해 A의 상태를 PCB에 저장하고 실행을 강제 중단한다.

4. 프로세스 B의 상태를 복원하고 CPU를 할당해 실행을 이어나간다.

#### 장단점

✅ **장점**

- 어떤 프로세스도 CPU를 무한정 독점할 수 없기에 웹 서버나 대화 시스템에서 클라이언트 요청에 즉각적인 응답을 보장할 수 있다.

- 모든 작업이 일정한 CPU 시간을 공평하게 나누어 가진다.

❎ **단점**

- 강제로 실행을 끊고 상태를 저장하고 복원하는 Context switching 비용이 시스템 전체 성능을 저하시킬 수 있다.

- 또한 여러 스레드가 동시에 공유 자원에 접근하다가 언제든 끊길 수 있어, Deadlock 이나 Race Condition 상태에 놓일 수 있다.

***


### 비선점형 스케줄링(Non-preemptive)

#### 📖 개념

- 한 프로세스가 CPU를 할당받으면 작업 종료 후 **CPU 반환 시까지 다른 프로세스는 CPU 점유가 불가한** 스케줄링 방식

#### 알고리즘

- 우선 순위, Deadline, FCFS(First Come First Served), SJF(Shortest Job First), HRN(Highest Response Ratio Next)

#### 메커니즘

1. 프로세스 A가 CPU를 **할당**받아 실행된다.

2. 다른 프로세스가 큐에 들어오더라도 A 의 작업이 끝날 때까지 계속 기다린다.

3. 프로세스 A 가 스스로 종료되거나 디스크 I/O를 요청해 `Blocked` 상태로 전환된다.

4. 이제 다른 프로세스 순서에 맞게 CPU를 할당한다.

#### 장단점

✅ **장점**

- 불필요한 Context Switchin이 발생하지 않아 CPU Throughput 을 효율적으로 관리할 수 있다.

- 프로세스가 자율적으로 자원을 제어하기에 공유 자원 동기화 문제가 상대적으로 적게 발생한다.

❎ **단점**

- 하나의 프로세스에서 문제가 발생하거나 무거운 배치 작업으로 인해 CPU 할당 시간이 계속해 길어지면 전체 시스템이 멈춘 것같은 문제가 발생한다. 

- 위를 기아 상태라 한다. 그래서 실시간 시스템에 부적합하다.
