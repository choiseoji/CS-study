## 자바의 객체가 저장되는 JVM 힙 영역과 메서드 실행 시 할당되는 가상 스택 영역이 실제 OS의 물리/가상 메모리에 어떻게 매핑되나요?

JVM의 Heap과 Stack 영역도 결국 운영체제가 JVM 프로세스에 할당한 가상 메모리 공간 위에 구성됩니다.

Heap 영역은 객체가 저장되는 공간으로, JVM이 운영체제로부터 큰 가상 메모리 영역을 할당받아 관리합니다. 실제 물리 메모리는 객체가 생성되어 해당 메모리에 접근하는 시점에 페이징을 통해 필요할 때 매핑됩니다.

반면 Stack 영역은 스레드마다 하나씩 생성되며, 메서드 호출 시 Stack Frame이 저장됩니다. Java Thread는 일반적으로 OS의 Native Thread와 1:1로 매핑되기 때문에, 각 스레드는 OS 레벨에서도 독립적인 Stack 메모리 영역을 가집니다.

즉 JVM은 Heap과 Stack이라는 논리적 메모리 구조를 관리하고, 실제 가상 메모리와 물리 메모리 매핑은 운영체제가 수행합니다.

</br>
</br>

### JVM 메모리 구조와 OS 매핑

```
Java 코드
   ↓
JVM 논리 메모리 (Heap, Stack, Metaspace ...)
   ↓
OS 가상 메모리 (Virtual Address Space)
   ↓
물리 메모리 (RAM + Swap)
```

JVM은 OS 위에서 그냥 프로세스 하나로 실행된다. OS는 JVM 내부의 Heap, Stack의 개념을 전혀 모르고, JVM이 메모리를 받아서 나눠쓰는 구조이다.

</br>

### **JVM Heap 매핑**

JVM이 시작될 때 OS에게 다음 두 조건으로 요청한다.

- `- Xmx` : OS에 예약만 하는 크기 (아직 Ram 할당 X)
- `- Xms` : JVM 시작 시 커밋되는 크기 (즉시 Ram 할당)

```
가상 주소 공간          물리 메모리
┌──────────────┐       ┌──────────┐
│  Reserved    │  ←X   │          │  예약만 됨, 매핑 없음
│  (Xmx)       │       │          │
├──────────────┤       ├──────────┤
│  Committed   │  ←──→ │ RAM Page │  실제 매핑됨
│  (Xms~현재)  │       │          │
└──────────────┘       └──────────┘
```

- 메모리가 부족해지면 GC를 먼저 수행하고, 그래도 부족하면 Reserved에서 조금씩 추가 커밋한다.
- Xmx까지 다 썼는데 부족하면 → OutOfMemoryError
- JVM은 이렇게 할당받은 메모리를 내부적으로 Eden, Survivor, Old Gen 등으로 나눠쓴다.

</br>

### **JVM Stack 매핑**

Heap은 JVM(GC)가 관리하지만, Stack은 OS가 스레드 단위로 직접 할당한다.

- 스레드 생성 시
    - Java 플랫폼 스레드는 OS 스레드와 1:1 대응
    - 스레드 생성 → OS가 `-Xss` 크기(기본 512KB~1MB)만큼 가상 주소 예약
    - 스택 맨 아래에 Guard Page 배치
- 메서드 호출 시
    - 메서드 호출 → SP(Stack Pointer)가 이동하면서 Stack Frame이 생성된다
    - 이 과정은 CPU 내부 동작이며 OS 개입은 없다
    - 메서드 호출 자체는 page fault와 무관한 CPU 동작이지만, 호출 과정에서 접근하는 메모리 페이지가 미커밋 상태라면 page fault는 발생할 수 있다.
    - 메서드 리턴 → SP가 내려가며 Frame이 제거된다 (GC와 무관하게 즉시 해제)
- Page Fault가 발생하는 경우
    - 스택이 아직 커밋되지 않은 메모리 영역으로 확장될 때
    - 또는 Guard Page를 침범하여 스택 오버플로우가 발생할 때
    - 이때만 OS가 물리 메모리를 할당하거나 예외(SIGSEGV)를 발생시킨다

```
프로세스 가상 주소 공간
┌─────────────────────────────────┐ 높은 주소
│  Thread-3 Stack  [256KB~1MB]    │ ← -Xss 설정값
├─────────────────────────────────┤
│  Guard Page (접근 시 SIGSEGV)   │ ← StackOverflowError 발생 지점
├─────────────────────────────────┤
│  Thread-2 Stack                 │
├─────────────────────────────────┤
│  Guard Page                     │
├─────────────────────────────────┤
│  Thread-1 Stack                 │
├─────────────────────────────────┤
│                                 │
│  JVM Heap                       │ ← mmap으로 확보한 큰 블록
│                                 │
├─────────────────────────────────┤
│  Metaspace (Native Memory)      │
├─────────────────────────────────┤
│  JVM 코드 자체 (.so/.dll)       │
└─────────────────────────────────┘ 낮은 주소
```

```
┌─────────────────┐ ← 스택 시작점 (높은 주소)
│  main() 프레임  │
├─────────────────┤
│  methodA() 프레임│
├─────────────────┤
│  methodB() 프레임│
├─────────────────┤  ← SP가 현재 여기
│  (빈 공간)      │
│                 │  ← 재귀호출 계속하면 SP가 계속 아래로
│                 │
├─────────────────┤
│   Guard Page    │  ← SP가 여기 닿는 순간 SIGSEGV
└─────────────────┘ 낮은 주소
```

### 🌟 Heap vs Stack 핵심 차이

**Heap**

- JVM이 OS으로부터 큰 메모리 영역을 확보한 뒤 GC가 내부에서 동적으로 관리하는 구조이다.
- 객체는 크기와 생성 위치가 불규칙하게 분포되기 때문에 메모리 접근이 비연속적이며, 이로 인해 새로운 페이지 접근이 자주 발생할 수 있다.

**Stack**

- 스레드 생성 시 OS가 고정 크기(-Xss)로 직접 할당하는 LIFO 구조의 메모리 영역이다.
- 메서드 호출과 반환에 따라 SP가 선형적으로 이동하기 때문에 메모리 사용 패턴이 매우 규칙적이며, 이미 커밋된 영역 내에서 동작하는 경우가 많아 page fault 발생 가능성이 낮다.
- LIFO 구조이기 때문에 별도의 GC가 필요 없고, 메서드 종료 시 자동으로 메모리가 해제된다.
- 스레드 수를 늘릴 때 Stack이 문제가 되는 이유 → 스레드 1000개를 만들면 Stack 공간만으로도 수백 MB가 즉시 예약됨
