## 데드락의 해결 방법 4가지(예방, 회피, 검출, 무시)를 간략히 설명해 주세요.
> 데드락 해결 방법은 예방, 회피, 검출 및 회복, 무시 4가지로 나눌 수 있습니다.
> 
> 예방은 상호 배제, 비선점, 점유 대기, 순환 대기 중 하나를 부정해서 데드락 발생 자체를 막는 방식입니다. 다만 자원 사용률이나 처리량이 떨어질 수 있습니다.
> 
> 회피는 자원 요청이 들어올 때마다 안전 상태인지 판단하고, 안전한 경우에만 자원을 할당하는 방식입니다. 대표적으로 은행원 알고리즘이 있지만, 최대 자원 요구량을 미리 알아야 해서 현실성은 낮습니다.
> 
> 검출은 데드락 발생을 허용하되 주기적으로 탐지하고, 발생하면 프로세스 종료나 자원 선점, 롤백으로 회복하는 방식입니다.
> 
> 마지막으로 무시는 데드락 가능성이 낮다고 보고 특별히 처리하지 않는 방식으로, 구현은 단순하지만 발생 시 시스템 중단 위험이 있습니다.

### 예방
교착 상태가 애초에 일어나지 않도록 방지하는 방법으로,

**교착상태의 발생조건 4가지 중 하나를 부정함**으로써 교착상태를 예방하는 방법

#### 1. 상호 배제 부정
여러 개의 프로세스가 동시에 공유 자원 사용 가능 (ex. 읽기 전용 파일)

-> 현실적으로 모든 자원에 적용할 수 없기에 상호 배제를 완전히 없애기 어려움

#### 2. 비선점 부정
- 모든 자원에 대한 선점을 허용
- 프로세스가 할당받을 수 없는 자원을 요청하는 경우, 기존에 가지고 있던 자원을 모두 반납하고 이전 자원을 얻기 위해 대기 (자원 낭비 발생)

-> 현질적으로 모든 자원에 적용 어려움 (ex. 프린터) / 선점 가능한 자원에 한해 강제 회수 허용

#### 3. 점유 대기 부정
- 프로세스 대기를 없애기 위해 프로세스 실행 전 필요 자원을 모두 할당 (자원 낭비 발생)
- 자원을 점유하고 있지 않을 때에만 다른 자원 요청 (기아 상태)

#### 4. 순환 대기 부정
자원을 선형으로 분류하여 고유 번호를 할당하고, 각 프로세스는 현재 점유한 자원의 고유번호보다 앞이나 뒤 한쪽 방향으로만 자원 요구 (자원 낭비 발생)

=> 시스템 처리량이나 자원 사용이 효율성을 떨어트리는 단점 존재

### 회피
교착 상태가 발생할 가능성이 있는 자원 할당을 하지 않고 안전한 상태에서만 자원 요청을 허용

=> 앞으로 데드락이 생길지 미리 게산

[회피를 위한 가정]
- **프로세스 수 고정**
  
    프로세스 수가 계속 바뀌면, 시스템이 계산해야 하는 대상이 계속 변화..!

    교착 상태 회피에서는 현재 존재하는 프로세스들이 각각
      
    ```
    얼마나 가지고 있는지
    앞으로 최대 얼마나 더 필요한지
    끝나려면 얼마나 더 받아야 하는지
    ```
    
    를 기준으로 안전 상태를 판단

    그런데 중간에 프로세스가 계속 추가되면, 기존에 “안전하다”고 판단했던 상태가 갑자기 안전하지 않을 수 있음!
    
    ex) 프로세스 3개 기준으로 자원 배분해도 모두 종료 가능 / 갑자기 새 프로세스 들어오면 전체 계산 달라짐
- **자원의 종류와 수 고정**
  
    ```
  현재 사용 중인 자원
    현재 남아 있는 자원
    각 프로세스가 추가로 필요한 자원
    ```
  이걸 비교해서 안전 상태 판단 
- **프로세스가 요구하는 최대 자원의 수 알아야 함**
  
    ```
  Need = Max - Allocation
  ```
  이걸 알아야 프로세스가 나중에 끝날 수 있는지 판단 가능

    이걸 몰면 앞으로 몇개가 더 필요할지 예측할 수 없기에 안전 상태 계산 어려움

- **프로세스는 자원 사용 후 반드시 반납**
  그래야... 다른 프로세스가 그 자원을 이어서 사용 가능 
  
### ‼️이런 가정들이 필요하기 때문에 현실성 부족 + 모든 자원 요청에 회피 알고리즘 계산은... 오버헤드

[회피 알고리즘]
- **은행원 알고리즘**
- **자원 할당 그래프 알고리즘**
    
    자원 유형마다 인스턴스가 하나 있는 경우 자원 할당 그래프 알고리즘을 사용해서 교착 상태 회피

    1. 자원 할당 그래프에 예약 간선 추가
  2. 프로세스 시작 전 모든 예약 간선들 자원 할당 그래프에 표시
  3. 프로세스는 예약 간선으로 설정한 자원에 대해서만 요청 가능, 주기 형성 되지 않을 때에만 자원 할당

### 탐지
데드락이 발생했는지에 대한 여부를 탐색하고 회복 기법 알고리즘 활용

=> 지속적으로 교착 상태를 확인하는 작업이 필요하기 때문에 오버헤드가 발생

### 회복
교착 상태가 발생했을 때 해결하는 기법

#### 1. 사용자 처리
교착 상태에 있는 프로세스 중 하나의 프로세스를 사용자가 강제 종료

#### 2. 시스템 처리
1. 프로세스 중지
   - 교착 상태에 속해 있는 모든 프로세스 중지
   - 교착 상태가 해결될 때까지 한 프로세스씩 중지
     - 희생자 프로세스 선택 (죽였을 때 손해가 가장 적고, 교착 상태를 빨리 풀 수 있는 프로세스)
2. 자원 선점
   - 프로세스들로부터 자원을 빼앗아 교착상태가 해결될 때까지 다른 프로세스들에게 자원을 할당
     - 선점 비용이 가장 낮고 교착 상태를 풀 가능성이 큰 자원부터 재할당
3. 롤백
    - 데드락이 발생할 것으로 예측되는 스레드들에 대해 주기적으로 상태를 저장했다가, 데드락이 발생하면 가장 최근에 저장해둔 상태로 복구

### 무시
데드락에 아무런 대비 없이 발생하도록 내버려 두는 방법

- 타조 알고리즘
  - 데드락에 아무 대비 없이 컴퓨터 시스템을 가동하고, 데드락이 발생하면 시스템을 재시작하거나 특정 스레드 강제 종료하는 방법으로 해결

### 정리
| 방법    | 핵심 아이디어                 | 장점               | 단점                            |
| ----- | ----------------------- | ---------------- | ----------------------------- |
| 예방    | 데드락 발생 조건 4가지 중 하나를 깨뜨림 | 데드락 발생 자체를 원천 차단 | 자원 사용률 저하, 시스템 처리량 감소         |
| 회피    | 안전 상태일 때만 자원 할당         | 예방보다 유연함         | 최대 자원 요구량 등을 미리 알아야 해서 현실성 낮음 |
| 검출/회복 | 일단 허용하고, 발생하면 찾아서 해결    | 자원 사용률이 비교적 좋음   | 탐지 비용과 복구 비용 발생               |
| 무시    | 데드락을 거의 고려하지 않음         | 구현이 단순하고 오버헤드 없음 | 실제 발생 시 시스템 중단 가능             |
