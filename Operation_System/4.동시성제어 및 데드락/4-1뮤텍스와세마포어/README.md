## 상호 배제를 위한 동기화 기법인 뮤텍스(Mutex)와 세마포어(Semaphore)의 개념과 결정적인 차이점은 무엇인가요?
>뮤텍스와 세마포어는 모두 공유 자원에 대한 동시 접근을 제어하기 위한 동기화 기법입니다. 
> 
> 뮤텍스는 하나의 자원에 하나의 스레드만 접근하도록 하는 소유권 기반 락이며, 락을 획득한 스레드만 해제할 수 있습니다. 
> 
>반면 세마포어는 카운터 값을 이용해 정해진 개수만큼 여러 스레드의 접근을 허용하는 방식입니다. 
> 
> 따라서 뮤텍스는 임계 구역 보호에 적합하고, 세마포어는 DB 커넥션 풀이나 외부 API 호출 제한처럼 동시에 사용할 수 있는 자원의 개수를 제어할 때 적합합니다.

### 동기화

여러 작업이 동시에 실행될 때 한 작업이 끝날 때까지 다른 작업이 시작되지 않는 것을 의미

### 뮤텍스
공유 자원에 대한 접근을 동시에 하나의 스레드만 가능하게 제한하는 방법

![뮤텍스](https://velog.velcdn.com/images/dodozee/post/9b2c8344-9348-4bea-89b1-6079accb7356/image.png)

- **동기화 대상이 한 개!**
- => 임계 구역 (Critical Section)을 가진 스레드들의 실행시간이 서로 겹치지 않고 각각 단독으로 실행되도록 하는 기술
- 한 프로세스에 의해 소유될 수 있는 Key 기반 상호배제 기법

  -> Key에 해당하는 객체가 있으며, 이 객체를 소유한 스레드만 공유 자원에 접근 가능

- 즉, 다중 프로세스들의 공유 리소스에 대한 접근을 조율하기 위해 동기화 또는 락을 사용함으로써 뮤텍스 객체를 두 스레드가 동시에 사용할 수 없음

### 세마포어
공유된 자원의 데이터 혹은 임계 영역 등에 여러 Process 혹은 Thread가 접근하는 것을 막아줌 (동기화 대상이 하나 이상)

![세마포어](https://img1.daumcdn.net/thumb/R1280x0/?scode=mtistory2&fname=https%3A%2F%2Fblog.kakaocdn.net%2Fdna%2FcYZOiu%2FbtrjvrzaimS%2FAAAAAAAAAAAAAAAAAAAAAHtr4M1GyNfHUSTY-xd5G07DB2a0TGELgm7mWdn5yBwT%2Fimg.png%3Fcredential%3DyqXZFxpELC7KVnFOS48ylbz2pIh7yKj8%26expires%3D1780239599%26allow_ip%3D%26allow_referer%3D%26signature%3D%252Bf4W7Vc2hb8YDFvn72YevcRv4RI%253D)

- 사용하고 있는 스레드/프로세스의 수를 공통으로 관리하는 하나의 값을 이용해 상호배제 달성
- 공유 자원에 접근할 수 있는 프로세스의 최대 허용치만큼 동시에 사용자가 접근할 수 있으며, 각 프로세스는 세마포어의 값을 확인하고 변경 가능
- 자원을 사용하지 않는 상태가 될 때, 대기하던 프로세스가 즉시 자원을 사용하고, 이미 다른 프로세스에 의해 사용 중이라는 사실을 알게되면 재시도 전에 일정시간 대기
- 비교적 긴 시간을 확보하는 리소스에 대해 사용

### 차이점
| 구분       | Mutex                     | Semaphore                     |
| -------- | ------------------------- | ----------------------------- |
| 사용 대상    | 동기화 대상이 **오직 1개**일 때 사용   | 동기화 대상이 **1개 이상**일 때 사용       |
| 자원 소유 여부 | 자원을 **소유할 수 있음**          | 자원을 **소유하지 않음**               |
| 책임       | Mutex를 소유한 스레드가 책임을 가짐    | 특정 스레드가 소유권을 갖지 않음            |
| 상태 값     | 상태가 **0 또는 1**뿐임          | 상태 값이 **0 이상의 정수 값**을 가질 수 있음 |
| Lock 여부  | Lock을 가질 수 있음             | Lock보다는 카운터 기반으로 동작           |
| 해제 가능 주체 | Mutex를 소유한 스레드만 해제 가능     | Semaphore를 소유하지 않은 스레드도 해제 가능 |
| 범위       | 일반적으로 **프로세스 범위**를 가짐     | 일반적으로 **시스템 범위**에 걸쳐 사용 가능    |
| 정리 방식    | 프로세스가 종료되면 자동으로 Clean up됨 | 파일 시스템 상의 파일 형태로 존재할 수 있음     |
| 대표 의미    | “한 번에 하나만 접근 가능”          | “정해진 개수만큼 접근 가능”              |

### 왜....?

Semaphore는 Mutex와 목적이 다르기 때문입니다.

Mutex는 주로 임계 구역 보호
```
공유 변수 수정
계좌 잔액 변경
재고 차감
구독 상태 변경
```

이런 작업은 “내가 잠갔으면 내가 풀어야” 안전

반면 Semaphore는 주로 자원 개수 제한에 씁니다.
```
DB 커넥션 최대 10개
외부 API 동시 호출 최대 5개
파일 처리 작업 최대 3개
동영상 인코딩 작업 최대 2개
```
이런 경우에는 “소유자”보다 현재 몇 개까지 사용 가능한가가 더 중요

## 추가 질문
### 세마포어가 카운터 기반이라고 하셨는데, 카운터를 증가/감소할 때 동시성 문제가 발생하지 않나요?
일반 변수처럼 카운터를 증가/감소하면 당연히 동시성 문제가 발생

그러나 세마포어 카운터 연산은 단순 연산이 아니고 원자적 역산

```text
wait(P)
- 카운터가 0보다 크면 카운터를 감소시키고 진입
- 카운터가 0이면 대기

signal(V)
- 작업이 끝나면 카운터를 증가
- 대기 중인 스레드가 있으면 하나를 깨움
```
카운터 확인 → 감소 → 대기 여부 판단 과정이 중간에 끊기지 않게 함

실제 세마포어 구현은 내부적으로 락, CAS, 커널의 원자적 명령어, 스케줄러의 대기 큐 등을 사용해서 카운터 변경을 안전하게 처리


### Monitor 방식
#### 세마포어의 단점
- 매번 임계 구역 앞뒤로 wait과 signal 함수 호출
- 세마포어를 누락하거나, 함수 호출 순서 헷갈린 경우, 중복해서 사용한 경우 예기치 못한 경과를 얻을 수 있음

#### 모니터 방식
![](https://img1.daumcdn.net/thumb/R1280x0/?scode=mtistory2&fname=https%3A%2F%2Fblog.kakaocdn.net%2Fdna%2FneIsQ%2FbtslxKuU2cS%2FAAAAAAAAAAAAAAAAAAAAANUJ2xctXwQc3sXG2s0RoqQkuwDRFhaeHsnhvuZAxLXv%2Fimg.png%3Fcredential%3DyqXZFxpELC7KVnFOS48ylbz2pIh7yKj8%26expires%3D1782831599%26allow_ip%3D%26allow_referer%3D%26signature%3Dl9VGmVfsDXCPno1LysC4KBh5kyA%253D)
공유 자원과 공유 자원에 접근하기 위한 인터페이스를 묶어서 관리

모니터를 통해 공유 자원에 접근하거자 하는 프로세스를 큐에 삽입하고, 큐에 삽입된 순서대로 하나씩 공유자원을 이용하도록 함

=> 모니터에 진입하기 위한 큐를 만들고, 모니터 안에 항상 하나의 프로세스만 들어오도록 하여 상호 배제를 위한 동기화 제공
