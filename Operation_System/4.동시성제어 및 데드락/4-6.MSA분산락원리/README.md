## 분산 인프라 환경(MSA)에서 단일 WAS의 자바 락 기법으로 동시성 제어가 불가능할 때, Redis나 ZooKeeper를 활용해 '분산 락'을 구현하는 원리는 무엇인가요?
>분산 락의 핵심 원리는 여러 WAS가 공유하는 외부 저장소에 락 상태를 기록해서, JVM이 달라도 동일한 자원에 대한 접근을 하나로 제한하는 것입니다.
> 
> 단일 WAS에서는 synchronized나 ReentrantLock이 하나의 JVM 메모리 안에서만 동작하기 때문에, MSA처럼 서버가 여러 대로 나뉘면 서버 A의 락 상태를 서버 B가 알 수 없습니다. 그래서 Redis나 ZooKeeper 같은 공통 저장소를 락 관리자로 두고, “이 자원은 현재 누가 사용 중인지”를 기록합니다.
> 
> 예를 들어 Redis는 SET key value NX PX 명령으로 락 키가 없을 때만 생성하게 하고, TTL을 걸어 서버가 죽어도 락이 영구적으로 남지 않게 합니다. 또한 unique value를 저장해서 락을 획득한 주체만 해제할 수 있게 합니다.
>
> ZooKeeper는 임시 순차 노드를 만들고, 가장 작은 순번의 노드를 가진 서버만 락을 획득하게 합니다. 세션이 끊기면 임시 노드가 삭제되기 때문에 장애 상황에서도 락 해제가 가능합니다.

### 왜 단일 서버와 MSA 환경이 차이가 날까?
#### 1. JVM 메모리 영역의 한계
- 단일 서버: 하나의 WAS(Tomcat 등) 안에서 하나의 JVM이 구동

  synchronized나 ReentrantLock은 동일한 JVM 내부의 힙(Heap) 메모리 영역을 공유하는 스레드들 사이에서만 상호 배제(Mutual Exclusion)를 보장

  하나의 똑같은 자바 객체를 두고 여러 스레드가 싸울 때 통제할 수 있는 수단

- 분산 환경: 각 서버마다 독립된 JVM과 독립된 메모리 공간

  서버 A의 JVM은 서버 B의 JVM 내부에서 어떤 스레드가 락을 획득했는지 전혀 알 방법이 없음!
  
#### 2. 분산 환경에서의 동시성 발생
- 클라이언트 1의 요청이 서버 A로 들어감. 서버 A의 JVM 1 내부에서 synchronized 블록을 통과해 락을 잡고 DB의 재고(현재 10개)를 읽어옴

- 동시에 클라이언트 2의 요청이 로드밸런서를 통해 서버 B로 들어감. 서버 B의 JVM 2 입장에서는 자기 내부에서 락을 잡은 스레드가 없으므로, synchronized 블록을 그대로 통과

- 결국 서버 A와 서버 B가 동시에 DB의 동일한 재고 데이터(10개)를 조회하게 되고, 각각 9개로 수정하는 쿼리를 날리면서 데이터 정합성이 깨지는(Race Condition) 문제가 발생

### 분산 락의 핵심 요구사항
- 상호 배제(Mutual Exclusion): 동시에 하나의 클라이언트만 락을 획득

- 데드락 방지: 락을 획득한 클라이언트가 비정상 종료되더라도 락이 영구적으로 유지 X

- 소유권 보장: 락을 획득한 클라이언트만 해당 락을 해제

### 분산 락을 구현하는 핵심 원리
여러 서버가 동시에 같은 자원에 접근하지 못하도록,

**공유 저장소에 “락을 잡은 주체”를 기록하고 그 기록을 기준으로 접근을 제어하는 것**

#### 1. Redis 분산 락
```redis
SET lock:order:1 unique-value NX PX 3000
```
| 옵션             | 의미             |
| -------------- | -------------- |
| `NX`           | 키가 없을 때만 저장    |
| `PX 3000`      | 3000ms 후 자동 만료 |
| `unique-value` | 락 소유자 식별값      |

lock:order:1이라는 키가 없으면 락 획득에 성공하고, 이미 있으면 다른 서버가 락을 잡고 있는 것으로 보고 실패

##### 락 흐름
```text
WAS-1 → Redis에 lock:order:1 생성 성공
       → 락 획득
       → 주문 처리

WAS-2 → Redis에 lock:order:1 생성 시도
       → 이미 키가 있음
       → 락 획득 실패 또는 대기

WAS-1 → 처리 완료 후 lock:order:1 삭제
       → 락 해제
```

- 만료 시간은 락을 잡은 서버가 죽으면 락 해제를 못할 수 있기에 이를 막기 위해 Redis에 TTL걸어 둠!

- unique value는 락을 잡은 서버가 락을 해제할 때, 자신이 잡은 락인지 확인하기 위해 사용

#### 2. ZooKeeper 분산 락
락 구현에 더 특화된 방식으로 사용

**[핵심]**
1. 임시 노드
2. 순차 노드

##### ZooKeeper 락 흐름
```text
/locks/order-1
```
특정 자원에 대한 락 경로 생성

```text
/locks/order-1/lock-00000001  ← WAS-1
/locks/order-1/lock-00000002  ← WAS-2
/locks/order-1/lock-00000003  ← WAS-3
```
여러 서버가 락을 잡으려고 하면 ZooKeeper에 순차 노드 생성

-> 가장 작은 번호를 가진 노드가 락을 획득

##### ZooKeepter 임시노드
시 노드는 해당 서버와 ZooKeeper의 세션이 끊기면 자동 삭제
```text
WAS-1 락 획득
WAS-1 장애 발생
ZooKeeper 세션 끊김
임시 노드 자동 삭제
WAS-2가 락 획득
```

Redis처럼 TTL 시간을 직접 정하는 방식보다 락 소유자의 생존 여부를 더 명확하게 관리

#### Redis vs ZooKeeper
| 구분      | Redis       | ZooKeeper           |
| ------- | ----------- | ------------------- |
| 락 획득 방식 | `SET NX PX` | 임시 순차 노드            |
| 락 해제    | 키 삭제        | 노드 삭제               |
| 장애 대응   | TTL 기반      | 세션 종료 시 임시 노드 삭제    |
| 성능      | 빠름          | Redis보다 무거움         |
| 구현 난이도  | 비교적 간단      | 상대적으로 복잡            |
| 사용 사례   | 짧고 빠른 락     | 강한 순서 보장, 안정성 중요한 락 |
