## 하드 링크와 심볼릭 링크의 차이가 뭔가요?

하드 링크는 원본 파일과 동일한 inode를 공유하는 또 다른 파일 이름입니다. 따라서 원본 파일을 삭제해도 다른 하드 링크가 남아 있으면 데이터에 계속 접근할 수 있습니다.

반면 심볼릭 링크는 원본 파일의 경로를 저장하는 특수 파일입니다. 원본이 삭제되거나 이동되면 링크가 끊어져 접근할 수 없게 됩니다.

또한 하드 링크는 일반적으로 같은 파일 시스템 내의 파일에만 생성할 수 있지만, 심볼릭 링크는 다른 파일 시스템이나 디렉터리도 가리킬 수 있습니다.

</br>
</br>

**파일 시스템 구조**

리눅스에서 파일은 크게 두 부분으로 구성된다.

```
파일 이름
    ↓
 inode
    ↓
실제 데이터
```

예를 들어,`cat test.txt` 를 실행하면 실제로는 아래 구조로 저장된다.

```
test.txt
    ↓
 inode 100
    ↓
실제 데이터
```

</br>

### **inode란?**

inode는 파일의 메타데이터와 실제 데이터 위치를 저장하는 파일 시스템의 관리 정보이다.

- 파일 크기
- 권한
- 생성 시간
- 데이터 블록 위치

반면 ‘파일 이름’은 inode에 저장되지 않는다.

파일 이름은 디렉터리가 관리하며, test.txt → inode 100 과 같은 정보가 디렉터리에 저장된다.

</br>

### 하드 링크 (Hard Link)

동일한 inode를 가리키는 새로운 파일 이름을 만드는 것이다.

예를 들어

```
ln test.txt hard.txt
```

를 실행하면 아래와 같은 구조가 된다.

```
test.txt ──┐
           ├── inode 100 ── 데이터
hard.txt ──┘
```

즉, inode 하나에 파일 이름이 두개인 상태가 된다.

**특징**

- 원본과 링크의 구분이 없다
    - 두 파일 모두 동일한 inode를 바라보기 때문에 사실상 같은 파일이다.
- 하나를 삭제해도 데이터는 남아있다
    - inode를 참조하는 링크가 하나라도 남아있다면 실제 데이터는 삭제되지 않는다.
    - 따라서 데이터를 완전히 삭제하려면 해당 inode를 참조하는 모든 하드 링크를 삭제해야 한다.
- 같은 파일 시스템에서만 생성 가능하다
    - 하드 링크는 동일한 inode를 공유해야 하므로 다른 파티션이나 다른 파일 시스템에는 생성할 수 없다.

</br>

### 심볼릭 링크 (Symbolic Link)

원본 파일의 경로를 저장하는 특수한 파일이다.

예를 들어,

```
ln -s test.txt sym.txt
```

을 실행하면 아래와 같은 구조가 된다.

```
sym.txt
   ↓
"test.txt"
   ↓
inode 100
   ↓
data
```

즉, inode를 직접 가리키는 것이 아니라 파일 경로를 저장하게 된다.

**특징**

- 원본 파일이 삭제되면 링크가 깨진다.
    - 가리키고 있는 파일이 삭제되었으니
    - Broken Symbolic Link
- 다른 파일 시스템도 가능하다
    - 심볼릭 링크는 단순히 경로 문자열을 저장하기 때문에 다른 디스크, 다른 파티션, 다른 파일 시스템도 자유롭게 연결할 수 있다.
- 디렉터리 링크도 가능하다

</br>

### 사용 사례

**WSL에서 Windows 파일 사용할 때**

실제 프로젝트가 Windows에 존재한다고 가정하자.

```
C:\Users\seoji\project
```

WSL에서는 다음 경로로 접근해야 한다.

```
/mnt/c/Users/seoji/project
```

하지만 매번 긴 경로를 입력하는 것은 번거롭다.

그래서 다음과 같이 심볼릭 링크를 생성할 수 있다.

```
ln-s /mnt/c/Users/seoji/project ~/project
```

그러면 아래와 같은 구조가 된다.

```
~/project
    ↓
/mnt/c/Users/seoji/project
```

이제 WSL에서는

```
cd ~/project
```

만 입력해도 실제 프로젝트에 접근할 수 있다.

코드를 복사하는 것이 아니라 원본 경로를 참조하므로 Windows와 WSL에서 동일한 소스를 사용할 수 있다.

</br>

**심볼릭 링크를 이용한 버전 전환 배포**

배포 파일이 다음과 같이 있다고 가정하자.

```
/app/releases/v1/myapp.jar
/app/releases/v2/myapp.jar
```

현재 서비스는 다음 심볼릭 링크를 사용한다.

```
/app/current
    ↓
/app/releases/v1
```

실행 스크립트는 항상 동일하다.

```
java-jar /app/current/myapp.jar

// 실제로는 아래 명령어가 실행됨
java-jar /app/releases/v1/myapp.jar
```

새 버전을 배포하고 싶다면 아래와 같이 심볼릭 링크만 변경하면 된다.

```
/app/current
    ↓
/app/releases/v2
```

그러면 동일한 실행 명령어가 v2꺼를 실행하게 된다.

```
java-jar /app/releases/v2/myapp.jar
```

이렇게 심볼릭 링크를 통해 배포를 하면 다음과 같은 장점이 있다.

- 실행 스크립트 수정 불필요
- 버전 전환이 쉬움
- 롤백이 간단함

</br>

**Q. rm은 파일을 지우는걸까?**

rm은 사실 파일을 지우는 명령어가 아니라, 디렉터리 엔드리를 제거하는 명령어이다.

```
inode 100
 ├── a.txt
 └── b.txt
 
link count = 2
```

이 상태일 때 rm a.txt를 한다고 파일이 지워지는게 아니고 다음과 같은 상태를 만드는거다.

```
inode 100
 └── b.txt
 
link count = 1
```

link count = 0이 되면 파일이 삭제된다.

</br>

**Q. 열려있는 파일을 삭제하면 어떻게 될까?**

```
tail -f app.log
```

로그 파일을 열어두고 rm app.log를 하면 로그 파일이 삭제될까?

아니다. 프로세스가 fd를 통해 inode를 잡고 있기 때문이다.

🚨 실무에서 발생 가능한 문제

운영 중 로그 파일이 너무 커져서 삭제했는데 df -h를 확인하면 디스크 사용량이 줄어들지 않는 경우가 있다.

이는 애플리케이션이 여전히 해당 파일을 열어둔 상태에서 로그를 계속 기록하고 있기 때문이다.

실제로는 파일명이 사라졌을 뿐, inode와 데이터 블록은 프로세스가 참조 중이므로 디스크 공간이 회수되지 않는다.

삭제된 상태로 열린 파일은 다음과 같이 확인할 수 있다.

```
lsof |grep deleted
```

그래서 실무에서는 rm을 하지 않고 로그 로테이션을 사용한다.

```
app.log → app.log.1 이름 변경
새로운 app.log 생성

---------------------
새로운 파일 : app.log
기존 파일 : app.log.1 (아직 java process가 얘랑 같은 inode 참조중)
```

예를 들어 현재 로그 파일이 app.log라면 app.log.1로 이름을 변경(rotate)한 뒤 새로운 app.log를 생성한다.

이후 애플리케이션이 새로운 app.log 파일을 다시 열도록(reopen) 하면, 기존 app.log.1에 대한 참조가 제거된다. 그 시점에는 더 이상 어떤 프로세스도 해당 inode를 사용하지 않으므로, app.log.1을 안전하게 압축하거나 삭제하여 디스크 공간을 회수할 수 있다.

이를 통해

- 로그 파일 크기 관리
- 디스크 Full 방지
- 서비스 중단 없는 로그 관리

가 가능하다. 실제 운영 환경에서는 보통 logrotate, Logback의 Rolling Policy, Log4j2의 Rolling File Appender 등을 이용해 이러한 과정을 자동화할 수 있다.

</br>

**Q. mv는 복사일까?**

같은 파일 시스템 내부에서 mv a.txt b.txt를 하는건 거의 rename()과 같은 역할을 한다

→ 데이터 복사 X, inode 그대로

하지만 `mv /mnt/disk1/a.txt /mnt/disk2/` 는 복사 + 삭제를 함 (파일 시스템이 다르기 때문에)

**Q.왜 디렉토리는 하드 링크를 금지할까?**

```
A
 └── B

B
 └── A
```

만약 가능하다면 이런 순환 구조가 가능하기 때문이다.
